# 데이터 설계와 검증 방법

## 1. 핵심 모델

장소 자체와 여정의 방문 기록을 분리한다. `cities.json`은 장소의 고정 정보, `journeys.json`의 `stops`는 특정 여정에서의 방문 순서와 사건을 담당한다. 이 구조는 루스드라·이고니온처럼 같은 여정에서 재방문한 도시와 여러 여정에서 반복 방문한 도시를 정확히 표현한다.

## 2. 신뢰도

- `explicit`: 본문이 장소, 이동 또는 사건을 직접 기록한다.
- `inferred`: 본문 흐름에서 합리적으로 추론되지만 이름이 직접 제시되지 않는다.
- `historical`: 고대 지명의 현대 비정, 연대, 서신 작성지 등 역사 연구에 의존한다.
- `disputed`: 후보 지점이나 해석이 여럿이며 합의가 충분하지 않다.

장소의 방문 사실이 explicit이어도 현대 좌표는 historical 또는 approximate일 수 있다. UI는 장소 전체 신뢰도와 stop 신뢰도를 함께 보여 준다.

## 3. 좌표 정책

좌표는 다음 중 하나의 대표점이다.

- 확인된 고대 유적 중심
- 일반적으로 받아들여지는 현대 도시 대응점
- 넓은 지역을 나타내는 지역 중심점
- 본문이 언급한 곶·해역을 설명하는 해상 웨이포인트

`precision`은 `approximate`, `regional` 등으로 표시한다. 좌표는 교육용 시각화 데이터이며 내비게이션·고고학 측량 데이터가 아니다.

## 4. 경로 정책

GeoJSON은 인접한 stop을 연결하는 선분이다. 본문이 이동 순서를 알려 주더라도 정확한 고대 도로와 해상 궤적을 항상 알려 주는 것은 아니므로 기본 선은 직선 근사다. `confidence !== explicit`인 구간은 점선으로 렌더링한다.

## 5. 보수적으로 처리한 사례

- `Acts 15:41`: 수리아와 길리기아의 개별 도시를 임의로 확정하지 않고 지역 노드를 사용했다.
- `Acts 16:6-8`: 갈라디아·브루기아·무시아는 지역 노드로 표시했다.
- `Acts 18:22`: “올라가 교회에 문안”을 예루살렘 방문으로 보는 일반적 해석을 사용하되 `inferred`로 표시했다.
- `Acts 20:2-3`: 헬라에서 석 달 머문 장소를 고린도로 확정하지 않고 아가야 지역 노드를 사용했다.
- `Acts 27:12`: 뵈닉스는 목표였으나 도착하지 못했으므로 route stop에서 제외하고 `unreached` 장소로 남겼다.
- `Acts 28:1`: 멜리데는 주류 견해인 몰타로 표시하지만 현대 비정은 `historical`로 구분했다.

## 6. 서신 연결

서신의 수신 교회처럼 명확한 관계와 작성 장소·연대 재구성을 구분한다. 예를 들어 고린도전서의 수신지는 고린도이지만 작성지를 에베소로 보는 것은 역사적 재구성이다. 갈라디아서와 남갈라디아 도시의 관계는 수신지 논쟁이 있으므로 `disputed`다.

## 7. 자동 검사

`scripts/validate-data.mjs`는 다음을 검사한다.

- 모든 ID의 고유성
- 도시의 국가·여정·인물 참조
- stop 순서와 이전/다음 링크
- 장절 필드 누락
- 이벤트 참조
- GeoJSON 선분 수와 stop 연결
- 좌표 형식

자동 검사는 신학적·역사적 판단을 대체하지 않는다. 데이터 변경 후에는 사도행전 본문 대조가 별도로 필요하다.
